# Repair The Candidate

Repair the candidate using the validation feedback.

Requirements:

- Fix schema and structural failures before geometry tuning.
- Fix build failures before interpreting baseline QC or authored checks.
- If validation reports `rounded_box_radius_too_large`, reduce that radius below half of the shortest size component or adjust the size; do not remove useful softened geometry just to pass.
- Treat baseline QC failures as harness-owned evidence.
- For overlap and isolation findings, decide whether the relationship is intentional. Fix unintended geometry; for fitted contacts add exact visual-pair bounded penetration proof; add scoped allowances only for other intentional exceptions.
- For exact check failures, preserve prompt-critical relationships and stable ids. Do not remove exact checks to hide failures.
- For sampled-pose failures, repair the mechanism's joint origin, axis, limits, child placement, or clearance at that pose; do not only tune the rest pose.
- For CAD-like mechanical assets, repair loose-looking mechanisms by adding or correcting fitted interfaces such as bearings, collars, guides, pins, liners, flanges, brackets, and exact relation checks. Do not pass validation by moving parts into visibly detached or barely tangent positions.
- If a complex mechanical candidate has many physical failures, treat it as a representation-level issue first. Simplify, split, shrink, or remove broad decorative static rails, walls, plates, covers, and shells that collide with the mechanism before making small per-pair tweaks.
- If overlaps involve a static cutaway housing, block, head, cover, guard, frame, rail, wall, case, or shell intersecting moving internals, open, split, shrink, move, or replace the static enclosure with windows, posts, collars, brackets, flanges, or open shell surfaces that clear the mechanism's swept volume. Do not add broad allowances or push shafts, rods, pistons, chains, gears, valves, or rotors into loose-looking positions.
- If overlaps are intentional inserted or captured mechanical fits, such as liners in housings, bushings in blocks, shafts in bearings, collars on hubs, sleeves in supports, or guide rails in slots, add or correct exact bounded `expect_within` or `expect_contact` proof for each repeated fitted pair rather than separating visually coupled parts.
- If validation reports missing mechanical relation coverage, add the correct authored evidence first: clearly named parts for requested routed paths or couplers; `expect_path_contacts` with exact path and target visual ids for routed belts, chains, tracks, wrapped cables, hoses, ropes, straps, and wires; exact relation checks at both ends for rods and linkages; and fitted-interface checks for pistons, sliders, shafts, gears, pulleys, sprockets, hubs, bearings, and wheels.
- If validation reports that a coupler needs rigid linkage geometry, replace connectorTube-only rods/linkages with rigid local capsule/tube/cylinder/bar geometry plus bearing-eye, clevis, pin, or socket end features. Keep exact endpoint checks; do not satisfy rigid rod quality with a flexible connector representation.
- If validation reports that a coupler needs a motion joint, add or correct a revolute pivot for the rod/linkage at a plausible pin, bearing-eye, clevis, socket, wrist-pin, crank-pin, or journal endpoint and bind it into the linked mechanism control. Do not silence the issue by making the rod fixed-only, deleting the coupler, or moving its checks to unrelated frame contacts.
- Do not satisfy wrapped path coverage with separate one-off `expect_contact` checks. Use one multi-target `expect_path_contacts` check whose `minContacts` covers the required supports.
- If validation says a prompt-critical rotary or path-support component is missing, add a clearly named part for the requested wheel, pulley, sprocket, gear, roller, rim, shaft, hub, bearing, or collar instead of relying on a visual label inside another part.
- For relation checks between mechanical parts with multiple visuals, reference the exact visual ids that carry the fit. Broad part-level checks are weak evidence and may not satisfy mechanical relation coverage.
- If repeated failures involve endpoint-routed cables, hoses, ropes, straps, tethers, bridge hangers, suspension cables, or wires, replace the rigid connector representation with `connectorTube` endpoint geometry instead of nudging static tube coordinates. For wrapped belts, chains, tracks, loops, or guided paths, reshape the path and satisfy `expect_path_contacts`.
- If overlaps involve a moving rotor/blade/wheel and a stationary grille, guard, cage, or shroud, move or resize the guard so it clears the swept volume; do not add broad overlap allowances between moving blades and protective bars.
- If repairs involve fan, compressor, turbine, propeller, or rotor blades, keep or add visible blade roots seated into a hub, drum, root band, collar, or dovetail mount. Do not create barely tangent blades just to clear overlap checks.
- Preserve or correct `controls` so linked joints share one dial only when they should move together; independent moving parts should remain independently controllable.
- When validation reports a prompt-critical rotary mover missing a motion joint, add or correct a revolute/continuous joint from a base, housing, support, bearing, collar, or hub to that crank, shaft, gear, pulley, sprocket, wheel, rotor, turbine, fan, or impeller.
- For linked mechanisms, repair the fitted interfaces across the whole authored control range. Validation samples intermediate control phases, so fix quarter-cycle and half-cycle rod, slider, belt, chain, gear, and shaft clearances instead of only moving the rest pose or endpoint.
- When validation reports missing controls, add manifest controls that cover every movable joint instead of deleting joints, changing movable joints to fixed, or relying on fallback dials for a multi-joint asset.
- Control `limits` accepts only numeric `lower` and `upper`; do not put `expect_*`, `part_exists`, `joint_exists`, or other check descriptors inside control limits.
- Control `joints` entries accept only control bindings `{ "jointId": "...", "scale": number, "offset": number }`; do not put `part_exists`, `joint_exists`, `expect_*`, `allow_*`, relation descriptors, or full joint definitions inside control joints. Add authored checks under `/checks/-`, allowances under `/allowances/-`, and joint definitions under `/joints`.
- When validation reports missing pose-specific mechanical relation coverage, add `check.pose` relation checks at a sampled driven pose for the same fitted interfaces: rod endpoints, guided movers in guides, and routed paths seated on their supports.
- For repeated linked mechanisms, repair the pose-specific relation coverage as one mechanism contract: reuse one named driven pose across the repeated rod endpoint checks, guided mover checks, and routed path checks that must stay coupled at the same phase.
- `check.pose` must be a compact joint pose: `{ "name": "...", "joints": [{ "jointId": "...", "value": number }] }`. Do not put transform fields such as `position`, `rotation`, or `scale` inside a check pose, do not put `joint_exists` objects inside `pose.joints`, do not nest another `expect_*` or check descriptor inside `pose`, and do not paste whole asset, parts, materials, checks, allowances, or metadata objects inside `pose`.
- Entries in `check.pose.joints` are sampled joint values, not full joint descriptors. Do not put `parentPartId`, `childPartId`, `origin`, `axis`, or `limits` inside pose samples.
- For material emission animation failures, repair the material `emissionAnimation` keyframe timing, on/off state, color, or intensity. Do not replace flashing material emission with separate glTF or scene light objects.
- For `surface_side_missing_check`, choose the intended material `side` and add `expect_material_side` for the exact visual. Use `double` when an open or paper-thin surface should be visible from both sides.
- For seated/captured fits, use exact visual-pair `expect_contact.maxPenetration`, `expect_path_contacts.maxPenetration`, `expect_gap.maxPenetration`, or `expect_within.maxPenetration` first so validation can treat the overlap as bounded fit proof. Use `expect_within.maxPenetration` for guided inner parts such as sliders in rails, pistons in liners, plungers in sleeves, and shafts in bearings. If an `allow_overlap` is still the right repair, scope it to the exact visual pair when possible and add or preserve exact proof checks for contact, path contact, bounded penetration, overlap, or containment.
- Use `maxPenetration: 0` only for actual no-penetration surface touch; use a small positive bound for captured bearings, collars, liners, sleeves, rails, slots, and seated wrapped paths.
- If validation reports `allowance_overlap_missing_proof_check`, add or correct a matching exact proof check for the same part pair and same visual pair; do not delete intentional-fit evidence unless the geometry no longer overlaps.
- If a failure repeats, reconsider the representation or support path instead of making another small tolerance or placement tweak.
- Do not remove, cap, fuse, or simplify prompt-critical visible geometry just to make validation pass.
- Return a focused JSON Patch object that repairs the supplied candidate while preserving unrelated stable ids and geometry.
- Prefer stable id-addressed patch paths for existing array items, such as `/parts/byId/deck-truss/visuals/byId/deck-panel/transform/position`, instead of brittle numeric array indices.
- Do not paste a complete Manifest3D asset object into a nested patch value such as joint `limits`, control `joints`, `check.pose`, visual `geometry`, or material fields. Patch only the exact nested property that should change.
- Keep patch values in the same schema domain as their path: visual `geometry` paths only accept primitive geometry descriptors, while `part_exists`, `joint_exists`, `expect_*`, and other authored checks belong under `/checks`, and `allow_*` descriptors belong under `/allowances`.
- Never write `allow_overlap`, `allow_isolated_part`, `allow_*`, or allowance fields such as `reason`, `partAId`, `partBId`, `visualAId`, or `visualBId` into a visual `geometry` field. Add or replace allowances under `/allowances`; replace visual `geometry` only with a valid primitive geometry descriptor.
- Do not use `part_exists` or `joint_exists` as a repair for physical overlap, fit, routed path, sampled-pose, clearance, or disconnection failures; those checks only prove ids exist.
- Do not return template, example, todo, placeholder, or dummy patch values. Every patch operation must use concrete current-candidate ids and values that solve a listed validation failure; if a partial repair is necessary, make a valid partial repair rather than a placeholder.
- Never use placeholder reference ids such as `x`, `y`, `z`, `a`, `b`, `todo`, `dummy`, `example`, `fake`, `placeholder`, `replace-me`, `__invalid__`, `invalid`, `invalid-id`, `part-a`, `part-b`, `visual-a`, `visual-b`, `joint-a`, or `joint-b`. Reuse existing stable ids from the supplied candidate, or add the real part/visual/joint before referencing it.
- When replacing vector fields, sizes, connector endpoint positions, or point arrays, write concrete numeric array values. Never use `[]` as a placeholder or deletion value; use `remove` only for array entries or object fields that should actually be deleted.
